## Frontmatter Field Reference

All fields are optional. Only `description` is recommended.

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `name` | string | No | directory name | Display name and `/slash-command`. Lowercase letters, numbers, hyphens only (max 64 chars). |
| `description` | string | Recommended | first paragraph of content | What the skill does and when to use it. Claude uses this for auto-invocation decisions. |
| `argument-hint` | string | No | — | Hint shown during autocomplete. Example: `[issue-number]` or `[filename] [format]`. |
| `disable-model-invocation` | boolean | No | `false` | `true` = only user can invoke via `/name`. Removes from Claude's context entirely. |
| `user-invocable` | boolean | No | `true` | `false` = hidden from `/` menu. Only Claude can invoke. Description stays in context. |
| `allowed-tools` | string (CSV) | No | — | Tools Claude can use without permission prompts when skill is active. |
| `model` | string | No | — | Model to use when skill is active. |
| `context` | string | No | — | Set to `fork` to run in a forked subagent context. |
| `agent` | string | No | `general-purpose` | Which subagent type to use when `context: fork` is set. Can be built-in (`Explore`, `Plan`, `general-purpose`) or custom agent name from `.claude/agents/`. |
| `hooks` | object | No | — | Hooks scoped to this skill's lifecycle. **WARNING: See gotchas.** |

### Invocation Matrix

| Frontmatter | User can invoke | Claude can invoke | Context loading |
|---|---|---|---|
| (default) | Yes | Yes | Description always in context; full SKILL.md loads when invoked |
| `disable-model-invocation: true` | Yes | No | Description NOT in context; loads only when user invokes |
| `user-invocable: false` | No | Yes | Description always in context; loads when Claude invokes |

**Note:** `user-invocable` only controls menu visibility, NOT Skill tool access. Use `disable-model-invocation: true` to block programmatic invocation.

## SKILL.md Structure

```
my-skill/
├── SKILL.md           # Main instructions (required)
├── template.md        # Template for Claude to fill in
├── examples/
│   └── sample.md      # Example output showing expected format
├── references/
│   └── detailed.md    # Reference docs loaded on demand
└── scripts/
    └── validate.sh    # Script Claude can execute
```

### Content Types

1. **Reference content** — Knowledge Claude applies to current work (conventions, patterns, domain knowledge). Runs inline.
2. **Task content** — Step-by-step instructions for a specific action. Often paired with `disable-model-invocation: true`.

**Keep SKILL.md under 500 lines.** Move detailed reference material to separate files.

### Supporting Files

Reference supporting files from SKILL.md so Claude knows when to load them:

```markdown
## Additional resources
- For complete API details, see [reference.md](reference.md)
- For usage examples, see [examples.md](examples.md)
```

Files are loaded on demand — they don't enter context until Claude reads them.

## String Substitutions

| Variable | Description |
|----------|-------------|
| `$ARGUMENTS` | All arguments passed when invoking. If not present in content, appended as `ARGUMENTS: <value>`. |
| `$ARGUMENTS[N]` / `$N` | Access specific argument by 0-based index. `$0` = first argument. |
| `${CLAUDE_SESSION_ID}` | Current session ID. |
| `${CLAUDE_SKILL_DIR}` | Directory containing the skill's SKILL.md. For plugin skills, the skill subdirectory (not plugin root). |

## Dynamic Context Injection

The `` !`command` `` syntax runs shell commands **before** skill content is sent to Claude. Output replaces the placeholder.

```yaml
---
name: pr-summary
description: Summarize changes in a pull request
context: fork
agent: Explore
---

## Pull request context
- PR diff: !`gh pr diff`
- PR comments: !`gh pr view --comments`
```

This is preprocessing — Claude only sees the final rendered content.

## Skill Locations & Priority

| Level | Path | Scope | Priority |
|-------|------|-------|----------|
| Enterprise | Managed settings | All org users | Highest |
| Personal | `~/.claude/skills/<name>/SKILL.md` | All your projects | — |
| Project | `.claude/skills/<name>/SKILL.md` | This project only | — |
| Plugin | `<plugin>/skills/<name>/SKILL.md` | Where plugin enabled | Namespaced (`plugin:skill`) |

When skills share the same name, higher-priority locations win. Plugin skills use `plugin-name:skill-name` namespace (no conflicts with other levels).

### Automatic Discovery

- Skills from nested `.claude/skills/` in subdirectories are discovered when editing files in those subdirectories (monorepo support).
- Skills from `--add-dir` directories are loaded automatically with live change detection.
- `.claude/commands/` files still work. If a skill and command share the same name, the skill takes precedence.

## Running in Subagents (context: fork)

```yaml
---
name: deep-research
context: fork
agent: Explore
---

Research $ARGUMENTS thoroughly:
1. Find relevant files
2. Analyze the code
3. Return findings
```

**How it works:**
1. New isolated context is created
2. Subagent receives SKILL.md content as its prompt
3. `agent` field determines execution environment (model, tools, permissions)
4. Results summarized and returned to main conversation

**WARNING:** `context: fork` only makes sense for skills with explicit task instructions. A skill with just guidelines (no actionable task) will return without meaningful output.

### Skills ↔ Subagents Relationship

| Approach | System prompt | Task | Also loads |
|---|---|---|---|
| Skill with `context: fork` | From agent type | SKILL.md content | CLAUDE.md |
| Subagent with `skills` field | Subagent's markdown body | Claude's delegation message | Preloaded skills + CLAUDE.md |

## Permission Control

Three ways to control skill access:

1. **Disable all skills**: Deny `Skill` tool in `/permissions`
2. **Allow/deny specific skills**: `Skill(commit)` (exact match), `Skill(deploy *)` (prefix match)
3. **Per-skill frontmatter**: `disable-model-invocation: true`

---

## Context Budget

Skill descriptions are loaded into context so Claude knows what's available. Budget:
- **2% of context window** (dynamic scaling)
- **Fallback: 16,000 characters**
- Check with `/context` for warnings about excluded skills
- Override: set `SLASH_COMMAND_TOOL_CHAR_BUDGET` environment variable

When subagents have preloaded skills (`skills` field in agent frontmatter), the **full skill content** is injected at startup (not just description).

### Gotchas

| Issue | Details | DOC-023 Ref |
|-------|---------|-------------|
| **Frontmatter hooks silently fail** | `hooks:` in skill frontmatter with `command:` field is accepted syntactically but never fires. Use settings.json hooks instead. | T10 |
| **No inline turn control** | Skills without `context: fork` run inline — there is no `maxTurns` for inline skills. For long-running work, always use `context: fork` + `agent:` with `maxTurns`. | Architecture Rule #1 |
| **`context: fork` shares filesystem** | Fork isolates conversation context only, NOT the filesystem. Forked agents can see all files from the main session. | T1-A9 |
| **Agent tool blocked in forks** | Forked agents CANNOT spawn subagents via Agent tool. Use Skill tool dispatch instead — it works at all depths. | T1-D2 |
| **`disallowedTools` inconsistent** | `disallowedTools: Bash` blocks Bash, but `disallowedTools: Edit` does NOT block Edit. Enforcement is tool-dependent. | T1-A5 |

## context: fork Behavior

When a skill uses `context: fork`, it runs in an isolated subagent context.

### What's Isolated
- Conversation context (separate context window)
- System prompt (from agent type)
- Tool restrictions (per agent config)

### What's Shared
- Filesystem (forked agent sees ALL files from main session) `[DOC-023 T1-A9]`
- CLAUDE.md files (loaded in fork too)

> **IMPORTANT:** "Fork" isolates conversation context, NOT the filesystem.

---

## Tool Restrictions

### tools Field
- Allowlist of tools the agent can use
- If omitted, inherits ALL tools from main conversation (including MCP tools)
- Use `/agents` UI for interactive tool selection

### disallowedTools Field
- Denylist — removed from inherited or specified tools

### Agent(agent_type) Restriction
- In `tools` field: `Agent(worker, researcher)` — only these agent types can be spawned
- Only applies to agents running as main thread with `claude --agent`
- Subagents CANNOT spawn other subagents (Agent tool blocked in forks)

### Confirmed Behavior (DOC-023)

| What | Works? | Ref |
|------|--------|-----|
| `disallowedTools: Bash` blocks Bash | Yes | T1-A4 |
| `disallowedTools: Edit` blocks Edit | **No** — Edit still works | T1-A5 |
| `tools` field as strict whitelist | No — appears to be a preload hint | T1 |
| Agent tool from within forks | **Blocked** | T1-D2 |
| Skill tool from within forks | Works at all depths | T1-B2, T9 |

> GOTCHA: `disallowedTools` enforcement is inconsistent. Bash blocks, Edit doesn't. `[undocumented]`

## Skill Dispatch from Agents

The **Skill tool is the universal dispatch mechanism** in forked contexts.

- Forked agents CAN use Skill tool to invoke other skills `[T1-B1/B2]`
- Forked agents CAN chain: agent -> skill (fork) -> child agent `[T1-C1-C4]`
- 3 levels of nested fork + Skill dispatch works end-to-end `[T9]`
- Each level gets own `maxTurns`, own agent persona, own tool restrictions `[T9]`
- Skill tool dispatch works at every depth level `[T9]`

> Architecture Rule: **Skill tool is the dispatch mechanism, not Agent tool.** Agent tool is blocked in forks.

---

## Nesting Patterns (DOC-023 T9: 3-Level Depth)

```
Main session
  └─ Skill (context:fork + agent:master)     [Level 1, own maxTurns]
       └─ Skill tool dispatch
            └─ Skill (context:fork + agent:sub-orch)  [Level 2, own maxTurns]
                 └─ Skill tool dispatch
                      └─ Skill (context:fork + agent:worker)  [Level 3, own maxTurns]
                           └─ reads references/, writes artifacts
```

- `references/` secret compartmentalization works at all depths `[T9]`
- Artifact-based coordination between levels works `[T9]`
- All coordination via `.shipkit/` artifacts — crash-recoverable `[T9]`

## Agent Teams

> **Experimental** — requires `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`

### Architecture

| Component | Role |
|-----------|------|
| **Team lead** | Main session that creates team, spawns teammates, coordinates |
| **Teammates** | Separate CC instances working on assigned tasks |
| **Task list** | Shared work items that teammates claim and complete |
| **Mailbox** | Messaging system for inter-agent communication |

### Team Primitives

- **TeamCreate** — Creates a team (lead only)
- **TaskCreate** — Creates tasks with descriptions and dependencies
- **TaskUpdate** — Updates task status
- **TaskList** — View all tasks with status
- **TaskGet** — Get full task details

### Task States
`pending` -> `in progress` -> `completed`

Tasks can have dependencies — blocked tasks unblock automatically when deps complete.

### Display Modes
- `"in-process"` — all teammates in main terminal (Shift+Down to cycle)
- `"tmux"` — each teammate in own split pane
- `"auto"` (default) — tmux if in tmux session, else in-process

### Quality Gate Hooks
- `TeammateIdle` — runs when teammate about to go idle. Exit code 2 = send feedback, keep working.
- `TaskCompleted` — runs when task being marked complete. Exit code 2 = prevent completion.

### Confirmed Behaviors (DOC-023)
- [x] TeamCreate + TaskCreate work from within forked agents `[T7]`
- [x] Teammates CAN invoke skills via Skill tool `[T7]`
- [x] Skills invoked by teammates produce correct artifacts `[T7]`
- [x] `Task(agent_type)` does NOT exist as a callable tool `[T8]`
- [x] Typing mechanism = teammate invokes skill with `context: fork` + `agent:` `[T8]`

### Limitations (Official)
- No session resumption for in-process teammates
- Task status can lag (teammates may not mark complete)
- One team per session
- No nested teams
- Lead is fixed (can't transfer)
- Permissions set at spawn time for all teammates
- Split panes require tmux or iTerm2

## Permission Modes for Subagents

| Mode | Behavior |
|------|----------|
| `default` | Standard permission checking with prompts |
| `acceptEdits` | Auto-accept file edits |
| `dontAsk` | Auto-deny prompts (explicitly allowed tools still work) |
| `bypassPermissions` | Skip ALL permission checks (use with caution) |
| `plan` | Read-only exploration mode |

## Background vs Foreground

- **Foreground** (default): blocks main conversation. Permission prompts passed through.
- **Background**: concurrent. Permissions pre-approved at launch. Auto-denies non-pre-approved.
- Set `background: true` in frontmatter or ask "run this in the background"
- Press **Ctrl+B** to background a running task
- Disable: `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1`

## Hook Event Types

| Event | When it fires | Matcher input | Can block (exit 2)? |
|-------|--------------|---------------|---------------------|
| `SessionStart` | Session begins or resumes | `startup`, `resume`, `clear`, `compact` | No |
| `UserPromptSubmit` | User submits prompt, before processing | (no matcher) | Yes — blocks prompt |
| `PreToolUse` | Before a tool call executes | tool name | Yes — blocks tool call |
| `PermissionRequest` | Permission dialog appears | tool name | Yes — denies permission |
| `PostToolUse` | After tool call succeeds | tool name | No — shows stderr to Claude |
| `PostToolUseFailure` | After tool call fails | tool name | No — shows stderr to Claude |
| `Notification` | CC sends a notification | notification type | No |
| `SubagentStart` | Subagent spawned | agent type name | No |
| `SubagentStop` | Subagent finishes | agent type name | Yes — prevents stopping |
| `Stop` | Claude finishes responding | (no matcher) | Yes — continues conversation |
| `TeammateIdle` | Teammate about to go idle | (no matcher) | Yes — keeps working |
| `TaskCompleted` | Task being marked complete | (no matcher) | Yes — prevents completion |
| `InstructionsLoaded` | CLAUDE.md or rules file loaded | (no matcher) | No |
| `ConfigChange` | Config file changes during session | config source | Yes — blocks change |
| `WorktreeCreate` | Worktree being created | (no matcher) | Yes — non-zero fails creation |
| `WorktreeRemove` | Worktree being removed | (no matcher) | No |
| `PreCompact` | Before context compaction | `manual`, `auto` | No |
| `SessionEnd` | Session terminates | termination reason | No |

### Matcher Values Reference

| Event | Matcher values |
|-------|---------------|
| Tool events | Tool names: `Bash`, `Edit`, `Write`, `Read`, `Grep`, `Glob`, `Skill`, `Agent`, `mcp__server__tool` |
| `SessionStart` | `startup`, `resume`, `clear`, `compact` |
| `SessionEnd` | `clear`, `logout`, `prompt_input_exit`, `bypass_permissions_disabled`, `other` |
| `Notification` | `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog` |
| `SubagentStart/Stop` | `Bash`, `Explore`, `Plan`, custom agent names |
| `ConfigChange` | `user_settings`, `project_settings`, `local_settings`, `policy_settings`, `skills` |
| `PreCompact` | `manual`, `auto` |

Matcher is a **regex** — `Edit|Write` matches either, `mcp__memory__.*` matches all memory server tools.

## Configuration Format

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/validate-bash.sh",
            "timeout": 600
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/lint.sh"
          }
        ]
      }
    ]
  }
}
```

### Three Nesting Levels
1. **Hook event** — lifecycle point (`PreToolUse`, `Stop`, etc.)
2. **Matcher group** — filter when it fires (`"Bash"`, `"Edit|Write"`)
3. **Hook handler(s)** — command/http/prompt/agent that runs

---

## Hook Handler Types

### Common Fields (all types)

| Field | Required | Default | Description |
|-------|----------|---------|-------------|
| `type` | Yes | — | `"command"`, `"http"`, `"prompt"`, or `"agent"` |
| `timeout` | No | 600s (cmd), 30s (prompt), 60s (agent) | Seconds before canceling |
| `statusMessage` | No | — | Custom spinner text while running |
| `once` | No | — | If `true`, runs once per session then removed (skills only) |

### Command Hook Fields

| Field | Required | Description |
|-------|----------|-------------|
| `command` | Yes | Shell command to execute |
| `async` | No | If `true`, runs in background without blocking |

### HTTP Hook Fields

| Field | Required | Description |
|-------|----------|-------------|
| `url` | Yes | URL for POST request |
| `headers` | No | Key-value headers. Values support `$VAR_NAME` interpolation |
| `allowedEnvVars` | No | Env var names allowed for header interpolation |

### Prompt Hook Fields

| Field | Required | Description |

## Exit Codes

| Code | Meaning | Behavior |
|------|---------|----------|
| **0** | Success | Proceed. stdout parsed for JSON output. |
| **2** | Block | Blocking error. stderr fed to Claude. Effect depends on event (see table above). |
| **Other** | Non-blocking error | stderr shown in verbose mode. Execution continues. |

**Important:** JSON output is ONLY processed on exit 0. Exit 2 ignores stdout entirely.

## JSON Output Fields

On exit 0, stdout can contain a JSON object:

| Field | Default | Description |
|-------|---------|-------------|
| `continue` | `true` | `false` = stop Claude entirely (overrides event decisions) |
| `stopReason` | — | Message to user when `continue: false` |
| `suppressOutput` | `false` | Hide stdout from verbose mode |
| `systemMessage` | — | Warning message shown to user |

### Decision Control Patterns

| Events | Pattern | Key fields |
|--------|---------|------------|
| UserPromptSubmit, PostToolUse, Stop, SubagentStop, ConfigChange | Top-level `decision` | `decision: "block"`, `reason` |
| TeammateIdle, TaskCompleted | Exit code 2 or `continue: false` | stderr feedback or stopReason |
| PreToolUse | `hookSpecificOutput` | `permissionDecision` (allow/deny/ask), `permissionDecisionReason` |
| PermissionRequest | `hookSpecificOutput` | `decision.behavior` (allow/deny), `updatedInput` |
| WorktreeCreate | stdout path | Print absolute path to created worktree |

## MCP Tool Matching

MCP tools follow pattern `mcp__<server>__<tool>`:

## Configuration Management

- Hooks are **snapshot at startup** — mid-session changes require review in `/hooks` menu
- `disableAllHooks: true` — temporarily disable all hooks
- `allowManagedHooksOnly: true` (managed only) — block user/project/plugin hooks
- `allowedHttpHookUrls` — URL allowlist for HTTP hooks (supports `*` wildcard)
- All matching hooks run in **parallel**; identical handlers deduplicated

## Gotchas

| Issue | Details | Ref |
|-------|---------|-----|
| **Frontmatter hooks silently fail** | `hooks:` in skill/agent frontmatter with `command:` is accepted but never fires. Use settings.json. | T10 |
| **Hooks snapshot at startup** | Changes to hook configs mid-session don't take effect until reviewed in `/hooks` | Official docs |
| **HTTP hooks can't block via status code** | Non-2xx = non-blocking error. Must return 2xx with JSON `decision: "block"` to block. | Official docs |
| **JSON only on exit 0** | Exit 2 ignores stdout entirely. Don't mix exit codes with JSON output. | Official docs |
| **Shell profile output breaks JSON** | If shell prints text on startup, it interferes with JSON parsing. | Official docs |

## Permission Rule Syntax

Format: `Tool` or `Tool(specifier)`

### Rule Evaluation Order
1. **Deny** rules evaluated first
2. **Ask** rules evaluated second
3. **Allow** rules evaluated third
4. **First matching rule wins**

### Tool-Specific Patterns

#### Bash
```
Bash                    — all commands
Bash(npm run *)         — prefix match with word boundary
Bash(npm*)              — prefix match without word boundary
Bash(git * main)        — wildcard in middle
Bash(* --version)       — wildcard at start
```

**Word boundary:** Space before `*` enforces it. `Bash(ls *)` matches `ls -la` but NOT `lsof`. `Bash(ls*)` matches both.

**Security:** CC is aware of shell operators (`&&`), so `Bash(safe-cmd *)` won't match `safe-cmd && other-cmd`.

#### Read/Edit
Follows gitignore spec:
```
Read(./.env)            — exact file in cwd
Read(./.env.*)          — pattern in cwd
Read(./secrets/**)      — recursive in cwd
Edit(/src/**/*.ts)      — relative to project root
Read(//Users/alice/*)   — absolute path (// prefix)
Read(~/Documents/*.pdf) — home directory
```

| Prefix | Meaning |
|--------|---------|
| `//` | Absolute filesystem root |
| `~/` | Home directory |
| `/` | Relative to project root |
| `./` or none | Relative to current directory |

`*` = single directory level, `**` = recursive.

#### WebFetch
```
WebFetch(domain:example.com)   — specific domain
WebFetch(domain:*.github.com)  — wildcard subdomain
WebFetch                       — all fetches
```

#### MCP
```
mcp__puppeteer                    — all tools from server
mcp__puppeteer__*                 — same (wildcard)
mcp__puppeteer__puppeteer_navigate — specific tool
MCP(github/search_repositories)   — alternative syntax
MCP(filesystem/*)                  — all tools from server
```

#### Agent (Subagents)
```
Agent(Explore)          — specific built-in agent
Agent(my-custom-agent)  — custom agent
Agent                   — all agents
```

#### Skill
```
Skill(commit)           — exact match
Skill(deploy *)         — prefix match with any arguments
Skill                   — all skills
```
